# Time Estimation Skill

You are a time estimation assistant. Analyze historical timing data and provide time predictions for planned tasks.

## 핵심 원칙: 시간의 80%는 출력물 생성에 쓰인다

실측 데이터에서 확인된 사실:
- **도구 호출 시간은 전체의 7~25%에 불과**
- 나머지 75~93%는 Claude의 **출력물 생성 시간** (HTML, 코드, Markdown 등)
- **"사고 시간"의 정체 = 출력 토큰 생성 시간** (생각과 쓰기가 동시에 일어남)
- 따라서 **"생성할 출력물의 예상 크기"**가 가장 정확한 예측 변수

### 예측 공식 (v3 — 출력물 기반)

```
예상 총 시간 = 출력물 생성 시간 + 도구 실행 시간 + 도구 간 판단 시간

출력물 생성 시간 = 예상 글자 수 × 글자당 생성 속도
도구 실행 시간   = 도구별 평균 × 호출 수
도구 간 판단 시간 = 도구 호출 수 × 평균 간격 (아래 판단 시간 표 참조)
```

### 판단 시간 보정 규칙 (v5 — 2026.04.16 실측 정밀 추적 반영)

Lab01~03 실측의 도구 간 간격(gap)을 정밀 분석한 결과:
- v4의 "연속 호출 3초"가 여전히 부정확 — **같은 도구라도 내용이 다르면 7~12초**
- 핵심 구분: **"단순 반복"** vs **"분석성 연속"**

#### 판단 시간 분류표

| 호출 패턴 | 판단 시간 | 예시 | 실측 근거 |
|-----------|----------|------|----------|
| **단순 반복** (동일 패턴) | **1~2초** | mkdir→ls, git status→git diff | Lab03: Bash→Bash 1초 |
| **병렬 호출** (동시 실행) | **0초** (1회만 카운트) | WebSearch×3 동시 | Lab02: WS→WS gap 포함 |
| **분석성 연속** (같은 도구, 다른 쿼리) | **8~12초** | grep(통계)→grep(감정)→grep(트렌드) | Lab01: Bash→Bash 평균 **11초** |
| **전환 호출** (다른 유형으로) | **5~7초** | Read→Edit, WebSearch→Write | Lab01: Read→Bash 6초 |
| **복잡한 판단** (분석→설계) | **12~15초** | 검색 결과 종합→리포트 구조 설계 | Lab02: WS→Write(분석 포함) 13초 |
| **대형 출력 생성** (→Write 직전) | **출력물 크기로 별도 계산** | 분석 결과→HTML Write | Lab01: 111초, Lab02: 92초 |

#### "분석성 연속" vs "단순 반복" 구분법

같은 도구(예: Bash)가 연속될 때, 다음 기준으로 분류:

| 질문 | "예"이면 | "아니오"이면 |
|------|---------|------------|
| 매번 다른 데이터를 분석하는가? | 분석성 연속 (8~12초) | 단순 반복 (1~2초) |
| 이전 결과를 보고 다음 쿼리를 결정하는가? | 분석성 연속 | 단순 반복 |
| 명령이 미리 다 정해져 있는가? | 단순 반복 | 분석성 연속 |

예시:
- `grep 에러` → `grep 팁` → `grep 트렌드` = **분석성 연속** (각 결과를 보고 다음 분석 방향 결정)
- `git status` → `git diff` → `git log` = **단순 반복** (미리 정해진 시퀀스)

#### 멀티 응답 전환 비용 (v5 추가)

`date` 기반 측정과 로그 기반 측정의 차이에서 발견:
Claude가 여러 번의 응답(response)으로 나뉘어 작업할 때, 응답 간 전환 비용이 발생.

| 상황 | 추가 비용 | 원인 |
|------|----------|------|
| 단일 응답으로 완료 | 0초 | 전환 없음 |
| 2~3개 응답으로 분할 | **15~30초/응답** | 컨텍스트 로드 + 사용자 승인 대기 |
| 권한 승인 팝업 포함 | **+5~15초/건** | 사용자 클릭 대기 |

예상 응답 수 = ceil(도구 호출 수 / 8) (1개 응답에 ~8개 도구 호출이 한계)

**적용 방법:**
1. 예상 도구 호출 시퀀스를 먼저 나열
2. 각 전환에 해당하는 판단 시간을 적용 (위 분류표 기준)
3. 병렬 호출은 1회로 카운트
4. 예상 응답 수를 산출하고 응답 간 전환 비용 추가

예) WebSearch×3(병렬) → 분석판단 → Write×1  (단일 응답)
= 판단 0초(병렬3개) + 12초(분석→설계) + 별도(HTML 생성 시간) = 12초 + 출력물 시간
(v3 공식: 4회 × 10초 = 40초 → 3배 과대)

예) Bash×8(분석성 연속) → Write×1  (단일 응답)
= 7회 전환 × 10초 + 별도(HTML 생성) = 70초 + 출력물 시간
(v3 공식: 9회 × 10초 = 90초 → 비슷하지만 구조가 다름)

### 사전 작업물 존재 확인 (v4 추가)

작업 시작 전, 관련 파일/결과물이 이미 존재하는지 확인할 것.
Lab03 실측에서 스킬이 이미 존재하여 예측 대비 77% 빠르게 완료된 사례 확인.

| 상황 | 보정 |
|------|------|
| 결과물이 전혀 없음 | 100% (보정 없음) |
| 유사한 결과물이 존재 (참고 가능) | **70%** (30% 할인) |
| 정확히 같은 결과물이 존재 (수정만 필요) | **40%** (60% 할인) |
| 중간 단계가 이미 완료됨 (예: 스킬 설치 완료) | **해당 단계 시간을 0으로** |

**적용 방법:**
1. 작업 시작 전 `ls`, `Glob`, `Read`로 기존 파일 확인
2. 해당하는 보정 비율을 전체 시간에 적용
3. 예측 결과에 "사전 작업물 보정" 항목으로 표시

### 측정 범위

이 시스템이 측정하는 것:
- `UserPromptSubmit` → `Stop`: 사용자가 Enter를 누른 시점부터 Claude 응답 완료까지
- `PreToolUse` → `PostToolUse`: 개별 도구 실행 시간
- 도구 간 간격: 출력물 생성 + 판단 시간

측정하지 않는 것:
- 사용자가 결과를 읽고 다음 명령을 입력하는 시간 (개발자 응답 시간)
- 즉, **순수 Claude 처리 시간만** 기록됨

주의: 도구 호출 시 권한 승인 대기가 있으면 해당 시간이 도구 시간에 포함될 수 있음.

## 환경 변수: 이 수치가 달라질 수 있는 조건들

아래 요인에 따라 실측 수치가 달라질 수 있으므로, 예측 시 현재 환경을 확인할 것.

### 크게 영향을 주는 요인 (2배 이상 차이 가능)

| 요인 | 영향 | 확인 방법 |
|------|------|----------|
| **모델** | Opus ~100 tok/s, Sonnet ~150 tok/s, Haiku ~200 tok/s. Sonnet은 Opus보다 ~1.5배 빠름 | 현재 세션의 모델 확인 |
| **컨텍스트 길이** | 대화가 길어질수록(50K+ 토큰) 출력 속도 저하. 극단적으로 2~3배 느려질 수 있음 | 대화 초반 vs 후반 비교 |
| **서버 부하 (피크 시간)** | 미국 업무 시간(한국 밤 11시~오전 8시)에 응답 속도 저하 가능 | 시간대 확인 |
| **Fast 모드** | 같은 Opus 모델이지만 출력이 빨라짐. 생성 시간 단축 | `/fast` 토글 여부 |

### 중간 영향 (30~50% 차이)

| 요인 | 영향 | 확인 방법 |
|------|------|----------|
| **출력 언어** | 한국어는 영어보다 토큰 효율이 낮음 (같은 의미에 더 많은 토큰). 한국어 출력이 ~30% 느릴 수 있음 | 출력 언어 확인 |
| **WebSearch 응답 시간** | 네트워크 환경에 따라 5~15초 편차 | 로그에서 WebSearch 평균 확인 |
| **병렬 도구 호출** | 2~3개 도구를 동시에 호출하면 개별 도구 시간은 같지만 전체 대기 시간 단축 | 로그에서 간격 0ms 패턴 확인 |

### 거의 영향 없는 요인 (10% 미만)

| 요인 | 이유 |
|------|------|
| **로컬 디스크 속도 (SSD/HDD)** | Read/Write는 0.3~0.5초로 전체의 1% 미만 |
| **OS (Windows/Mac/Linux)** | 도구 실행 시간 차이 미미, 사고 시간은 서버 측 |
| **프로젝트 크기** | 도구 호출 수에는 영향, 글자당 생성 속도에는 무관 |

### 환경 보정 가이드

예측 시 다음 보정을 적용:
```
보정된 시간 = 기본 예측 × 모델 계수 × 컨텍스트 계수 × 언어 계수

모델 계수:    Opus=1.0, Sonnet=0.67, Haiku=0.5
컨텍스트 계수: 대화 초반(~30K)=1.0, 중반(~100K)=1.1, 후반(200K+)=1.3, 극후반(500K+)=1.5  # v4: 1M 모델 기준 상향
언어 계수:    영어=1.0, 한국어=1.3, 혼합=1.15
```

## Step 0: 캘리브레이션 데이터 확인

예측 전에 **반드시** 실측 캘리브레이션 데이터를 먼저 확인한다:

```bash
bash ~/.claude/scripts/calibrate.sh  # calibration.json 생성/갱신
cat ~/.claude/timing/calibration.json
```

### calibration.json 구조

```json
{
  "calibration_version": 1,
  "sample_count": { "tools": 847, "responses": 42 },
  "tool_avg": {
    "Read": { "avg_ms": 480, "min_ms": 200, "max_ms": 1200, "count": 150 },
    "Bash": { "avg_ms": 2300, ... },
    ...
  },
  "gap": {
    "same_tool": { "avg_ms": 5200, "count": 80 },
    "diff_tool": { "avg_ms": 6800, "count": 120 },
    "overall":   { "avg_ms": 6100, "count": 200 }
  },
  "gap_histogram": { "0-1s": 12, "1-3s": 45, "3-5s": 30, ... },
  "response_avg": { "avg_duration_ms": 45000, "avg_tools_per_response": 5.2 }
}
```

### 실측값 우선 원칙

| 항목 | calibration.json 있을 때 | 없을 때 |
|------|------------------------|---------|
| 도구 실행 시간 | `tool_avg.{도구}.avg_ms` 사용 | 아래 기본값 사용 |
| 판단 시간 (gap) | `gap.same_tool.avg_ms` / `gap.diff_tool.avg_ms` 사용 | v5 분류표 기본값 사용 |
| 응답당 시간 | `response_avg.avg_duration_ms` 참고 | 계산으로 추정 |

**sample_count.tools가 20 미만이면** 기본값과 실측값을 가중평균한다:
```
혼합값 = 기본값 × (1 - n/20) + 실측값 × (n/20)    (n = sample count)
```


## Step 2: 시간 예측 (작업 설명이 있는 경우)

`$ARGUMENTS`가 제공되면:

### 2-1. 작업 분석

작업을 분석하여:
1. **캘리브레이션 데이터 로드** — calibration.json이 있으면 실측값 우선
2. **생성할 출력물** 식별 — HTML? 코드? 설정 파일? 몇 개?

### 2-2. 예측 계산

```
① 출력물 생성 시간:
   각 출력물의 (예상 글자 수 × 글자당 속도) 합산

② 도구 실행 시간:
   calibration.json의 tool_avg가 있으면 실측값 사용, 없으면 아래 기본값:
   - Read: ~0.5초, Write: ~0.4초, Edit: ~0.4초
   - Bash: ~1~5초 (명령에 따라 가변)
   - Grep/Glob: ~1초
   - WebSearch: ~9초
   - Agent: ~30~60초

③ 도구 간 판단 시간:
   calibration.json의 gap 데이터가 있으면 실측값 사용:
   - same_tool.avg_ms → 같은 도구 연속 시 gap
   - diff_tool.avg_ms → 다른 도구 전환 시 gap
   실측값이 없으면 v5 분류표 기본값 적용:
   - 단순 반복: 1~2초, 분석성 연속: 8~12초, 병렬: 0초, 전환: 5~7초, 복잡판단: 12~15초
   - "→Write 직전"의 대형 출력 생성은 ①의 출력물 생성 시간으로 별도 계산

③-b 멀티 응답 전환 비용:
   예상 응답 수 = ceil(도구 호출 수 / 8)
   추가 비용 = (응답 수 - 1) × 20초

③-c 사전 작업물 보정:
   기존 결과물 존재 여부를 확인하여 할인 적용 (v4 사전 작업물 표 참조)

④ 환경 보정:
   (① + ② + ③) × 모델 계수 × 컨텍스트 계수 × 언어 계수
```

## 인라인 예측 모드 (CLAUDE.md 연동)

`/estimate`로 명시적 호출하는 것 외에, CLAUDE.md에 아래 지침을 추가하면 **모든 응답의 첫 줄에** 예상 시간을 자동 표시할 수 있습니다.

### CLAUDE.md 권장 설정

```markdown
## 작업 시간 예측 자동 표시

**모든 응답의 첫 줄에** 예상 소요 시간을 표시하세요. 구현 작업뿐 아니라 질의응답, 탐색, 설명 등 모든 유형의 요청에 적용합니다. TaskCreate/TaskUpdate 사용 여부와 무관하게 항상 표시합니다.

estimate 스킬(v5)의 공식을 적용합니다:

1. 생성할 출력물의 예상 크기(글자 수)와 유형을 파악
2. 필요한 도구 호출 시퀀스를 나열
3. 판단 시간 분류표(v5)에 따라 각 전환별 시간 차등 적용:
   - 단순 반복: 1~2초 / 분석성 연속: 8~12초 / 병렬: 0초 / 전환: 5~7초 / 복잡판단: 12~15초
4. 사전 작업물 존재 여부 확인 후 할인 적용
5. 환경 보정 적용 (모델 계수 x 컨텍스트 계수 x 언어 계수)

표시 형식 (간결하게 한 줄):
\`\`\`
> 예상: ~N분 M초 (출력물 Xs + 도구 Ys + 판단 Zs | 보정: 모델×컨텍스트×언어)
\`\`\`
```

### 인라인 vs 상세 모드 구분

| 모드 | 트리거 | 출력 형식 |
|------|--------|----------|
| **인라인** | CLAUDE.md에 의한 자동 표시 (모든 응답) | 한 줄: `> 예상: ~N분 M초 (...)` |
| **상세** | `/estimate` 명시적 호출 | 전체 테이블 (출력물, 도구, 판단 시간 상세) |

인라인 모드에서는 동일한 v5 공식을 적용하되, 결과를 한 줄로 압축합니다. 상세 내역이 필요하면 `/estimate`를 호출하면 됩니다.
