# Time Estimation Skill

You are a time estimation assistant. Analyze historical timing data and provide time predictions for planned tasks.

## 핵심 원칙: 시간의 80%는 출력물 생성에 쓰인다

실측 데이터에서 확인된 사실:
- **도구 호출 시간은 전체의 7~25%에 불과**
- 나머지 75~93%는 Claude의 **출력물 생성 시간** (HTML, 코드, Markdown 등)
- **"사고 시간"의 정체 = 출력 토큰 생성 시간** (생각과 쓰기가 동시에 일어남)
- 따라서 **"생성할 출력물의 예상 크기"**가 가장 정확한 예측 변수

### 예측 공식 (v3 — 출력물 기반)

```
예상 총 시간 = 출력물 생성 시간 + 도구 실행 시간 + 도구 간 판단 시간

출력물 생성 시간 = 예상 글자 수 × 글자당 생성 속도
도구 실행 시간   = 도구별 평균 × 호출 수
도구 간 판단 시간 = 도구 호출 수 × 평균 간격 (짧은 사고, ~10초)
```

### 출력물 유형별 생성 속도 (실측 기준)

| 출력물 유형 | 글자당 속도 | 이유 | 실측 근거 |
|------------|-----------|------|----------|
| HTML (구조화+한국어) | **~7ms/글자** | 태그가 반복 패턴, 토큰 효율 높음 | Lab01: 16,195글자→111초, Lab02: 13,261글자→101초 |
| Markdown / 설정 파일 | **~15ms/글자** | 매 줄 새로운 판단 필요 | Lab03: 2,063글자→33초, estimate: 3,520글자→49초 |
| 셸 스크립트 / 코드 | **~15ms/글자** | 로직 판단 비용 높음 | timing.sh: 1,835글자→29초 |
| 짧은 Edit (diff) | **~20ms/글자** | 전후 맥락 판단 비용 | 추정 |

### 작업 유형별 예상 출력 크기

| 작업 유형 | 예상 글자 수 | 예상 생성 시간 |
|-----------|------------|-------------|
| HTML 리포트 (풀) | 13,000~16,000 | 90~110초 |
| HTML 리포트 (간단) | 5,000~8,000 | 35~55초 |
| SKILL.md / 설정 파일 | 2,000~3,500 | 30~50초 |
| 셸 스크립트 | 1,000~2,000 | 15~30초 |
| 코드 수정 (Edit) | 100~500 | 5~10초 |

### 측정 범위

이 시스템이 측정하는 것:
- `UserPromptSubmit` → `Stop`: 사용자가 Enter를 누른 시점부터 Claude 응답 완료까지
- `PreToolUse` → `PostToolUse`: 개별 도구 실행 시간
- 도구 간 간격: 출력물 생성 + 판단 시간

측정하지 않는 것:
- 사용자가 결과를 읽고 다음 명령을 입력하는 시간 (개발자 응답 시간)
- 즉, **순수 Claude 처리 시간만** 기록됨

주의: 도구 호출 시 권한 승인 대기가 있으면 해당 시간이 도구 시간에 포함될 수 있음.

## 환경 변수: 이 수치가 달라질 수 있는 조건들

아래 요인에 따라 실측 수치가 달라질 수 있으므로, 예측 시 현재 환경을 확인할 것.

### 크게 영향을 주는 요인 (2배 이상 차이 가능)

| 요인 | 영향 | 확인 방법 |
|------|------|----------|
| **모델** | Opus ~100 tok/s, Sonnet ~150 tok/s, Haiku ~200 tok/s. Sonnet은 Opus보다 ~1.5배 빠름 | 현재 세션의 모델 확인 |
| **컨텍스트 길이** | 대화가 길어질수록(50K+ 토큰) 출력 속도 저하. 극단적으로 2~3배 느려질 수 있음 | 대화 초반 vs 후반 비교 |
| **서버 부하 (피크 시간)** | 미국 업무 시간(한국 밤 11시~오전 8시)에 응답 속도 저하 가능 | 시간대 확인 |
| **Fast 모드** | 같은 Opus 모델이지만 출력이 빨라짐. 생성 시간 단축 | `/fast` 토글 여부 |

### 중간 영향 (30~50% 차이)

| 요인 | 영향 | 확인 방법 |
|------|------|----------|
| **출력 언어** | 한국어는 영어보다 토큰 효율이 낮음 (같은 의미에 더 많은 토큰). 한국어 출력이 ~30% 느릴 수 있음 | 출력 언어 확인 |
| **WebSearch 응답 시간** | 네트워크 환경에 따라 5~15초 편차 | 로그에서 WebSearch 평균 확인 |
| **병렬 도구 호출** | 2~3개 도구를 동시에 호출하면 개별 도구 시간은 같지만 전체 대기 시간 단축 | 로그에서 간격 0ms 패턴 확인 |

### 거의 영향 없는 요인 (10% 미만)

| 요인 | 이유 |
|------|------|
| **로컬 디스크 속도 (SSD/HDD)** | Read/Write는 0.3~0.5초로 전체의 1% 미만 |
| **OS (Windows/Mac/Linux)** | 도구 실행 시간 차이 미미, 사고 시간은 서버 측 |
| **프로젝트 크기** | 도구 호출 수에는 영향, 글자당 생성 속도에는 무관 |

### 환경 보정 가이드

예측 시 다음 보정을 적용:
```
보정된 시간 = 기본 예측 × 모델 계수 × 컨텍스트 계수 × 언어 계수

모델 계수:    Opus=1.0, Sonnet=0.67, Haiku=0.5
컨텍스트 계수: 대화 초반(~10K)=1.0, 중반(~50K)=1.2, 후반(100K+)=1.5
언어 계수:    영어=1.0, 한국어=1.3, 혼합=1.15
```

## Step 1: 타이밍 데이터 읽기

```bash
bash ~/.claude/scripts/timing-stats.sh
```

타이밍 로그(`~/.claude/timing/timing.log`) 형식:
```
timestamp_ms|type|name|duration_ms
```

- `type=tool`: 개별 도구 호출. `name`은 도구명
- `type=response`: 전체 응답. `name`은 `tools:N` (N = 해당 응답의 도구 호출 수)
- `type=lab_start/lab_end`: 작업 단위 시작/종료 마커

## Step 2: 시간 예측 (작업 설명이 있는 경우)

`$ARGUMENTS`가 제공되면:

### 2-1. 작업 분석

작업을 분석하여:
1. **생성할 출력물** 식별 — HTML? 코드? 설정 파일? 몇 개?
2. **필요한 도구 호출** 예상 — Read, Grep, Bash, WebSearch 등 각 몇 회?
3. **현재 환경** 확인 — 모델, 대화 길이, 시간대

### 2-2. 예측 계산

```
① 출력물 생성 시간:
   각 출력물의 (예상 글자 수 × 글자당 속도) 합산

② 도구 실행 시간:
   각 도구의 (평균 시간 × 호출 수) 합산
   - Read: ~0.5초, Write: ~0.4초, Edit: ~0.4초
   - Bash: ~1~5초 (명령에 따라 가변)
   - Grep/Glob: ~1초
   - WebSearch: ~9초
   - Agent: ~30~60초

③ 도구 간 판단 시간:
   도구 호출 수 × ~10초 (다음 행동 결정)

④ 환경 보정:
   (① + ② + ③) × 모델 계수 × 컨텍스트 계수 × 언어 계수
```

### 2-3. 결과 출력

```
## ⏱ 예상 소요 시간

### 출력물 생성
| 출력물 | 예상 크기 | 유형 | 예상 시간 |
|--------|---------|------|----------|
| report.html | ~14,000글자 | HTML | ~100초 |

### 도구 실행
| 도구 | 호출 수 | 예상 시간 |
|------|--------|----------|
| WebSearch | x3 | ~27초 |
| Read | x2 | ~1초 |
| Bash | x3 | ~5초 |
| Write | x1 | ~0.4초 |

### 합산
| 항목 | 시간 |
|------|------|
| 출력물 생성 | ~100초 |
| 도구 실행 | ~33초 |
| 판단 시간 (9회 × 10초) | ~90초 |
| **소계** | **~223초** |
| 환경 보정 (Opus, 한국어, 대화 중반) | ×1.0 ×1.3 ×1.2 |
| **최종 예측** | **~4분 30초** |

> 📊 데이터 N회 기반 | 신뢰 구간: ±30%
> 환경: Opus 4.6, 한국어, 대화 중반
```

## Step 3: 통계 요약 (작업 설명이 없는 경우)

타이밍 통계를 보여줄 때는 다음을 포함:
- 도구별 순수 실행 시간
- 도구 간 사고 시간 분포 (0-1초, 1-5초, 5-10초, 10-20초, 20-50초, 50초+)
- 50초 이상 사고 구간 상세 (병목 지점)
- 출력물 크기 vs 사고 시간 상관관계 (Write된 파일이 있으면)
- 실측 Lab 데이터 (있으면)

## Important Notes

- timing.log가 없거나 비어있으면, 데이터 축적이 필요하다고 안내. 20~30회 도구 호출 후 다시 확인 권장.
- 밀리초를 사람이 읽기 좋은 형식으로 변환 (초/분).
- 적절한 정밀도로 반올림 ("12.847초" → "~13초").
- 불확실성에 솔직하게 — 데이터가 많을수록 정확도 향상.
- 실측 데이터가 현재 스킬의 기본값과 크게 다르면, 실측 데이터를 우선한다.
